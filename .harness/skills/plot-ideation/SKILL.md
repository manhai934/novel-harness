# plot-ideation · 创作灵感 skill

> 本模块是 `novel-harness` 的**创作子模块**。
> 职责：当作者卡文、需要新剧情、或想探索不同故事走向时，提供多方向的创作灵感。
> 
> **核心理念**：创作 Agent 只管"出点子"，审稿 Agent 负责"挑毛病"。
> 做事的和评判的分开——发散时不自我审查，审查时不发散。

---

## 1. 触发条件

以下情况应加载本 skill：

- **卡文求助**：用户说"不知道接下来怎么写""卡文了"
- **剧情推演**：用户说"如果xxx会怎样""帮我推演一下剧情走向"
- **设定扩展**：用户说"想加个新地点/新角色/新组织/新规则/新道具"但没具体想法
- **反转/钩子设计**：用户说"这章结尾加个悬念""帮我设计个反转"
- **备选方案**：用户说"有几个方向拿不准，帮我对比一下"
- **日常灵感**：用户没有具体问题，但想闲聊式碰撞想法
- **灵感整理**：用户给了很多零散想法，想先结构化归档和排序

**不适用**：用户已经明确知道要写什么、只是需要执行时。

---

## 2. 执行流程

### 2.1 理解上下文

无论哪种灵感请求，第一步都是理解当前故事状态：

```
加载：
  □ 最新章节的正文或摘要（最近 1-3 章）
  □ 主要角色的当前状态（身体/能力/关系/处境/位置/目标）
  □ 当前已有的伏笔和未回收的线索
  □ 可用资源（队友/道具/信息差）
  □ 世界观约束（什么能做、什么不能做）

产出：context_summary.md
  ── 故事当前位置
  ── 角色状态
  ── 可用伏笔
  ── 世界观边界（不能打破的约束）
```

**关键**：如果用户没有提供上下文，主动问清楚再开始。

**题材隔离**：本 skill 是通用灵感模块。不得默认把故事理解成游戏、系统、数据化或网游题材；`副本/面板/等级/装备/技能/玩家/系统提示` 只在当前项目或用户请求明确属于该题材时使用。

如果当前任务重点是“把零散灵感整理成可比较的方向”，加载 `references/公共灵感整理与组合.md`。
如果当前任务已经接近“准备写这一章了”，加载 `references/章节写前准备清单.md`。

### 2.2 判断灵感类型

根据用户需求，进入对应模块：

```
用户说：
  "接下来怎么写？"          → 模块A：剧情构思
  "想加个新设定"            → 模块B：设定扩展
  "章尾加个悬念"            → 模块C：反转/钩子设计
  "有几个方向拿不准"        → 模块A + 模块C（先发散，再配钩子）
  模糊请求                  → 先按模块A走，同时问用户偏好
```

---

### 模块A：剧情构思

**目标**：基于当前故事状态，生成多条可能的剧情走向。

**执行步骤**：

```
Step 1: 识别可用资源
  当前冲突是什么？角色有什么未完成的目标？有什么可用伏笔？
  
Step 2: 发散构思（产出 3-5 个方向）
  每个方向包含：
  □ 一句话核心假设（如果……会怎样）
  □ 灵感类型（题材/人设/世界观/钩子/情绪体验等）
  □ 情感内核（读者到底会因为什么继续看）
  □ 题材方向标签（按当前项目题材选择；游戏/数据化题材才参考 game-datafied）
  □ 核心冲突/爽点类型
  □ 本方向的核心玩法（追查/关系拉扯/求生资源/经营扩张/修炼突破/竞技对抗等）
  □ 预计篇幅（短期 1-3 章 / 中期 5-10 章 / 长期 10+ 章）
  □ 所需的新角色/场景（如果有）
  □ 不可越界内容（本方向不写什么，避免发散）
  □ 风险提示（会不会破坏平衡/会不会拖节奏）
  
Step 3: 对比呈现
  方向A vs 方向B vs 方向C：
  ┌─────────┬──────────┬──────────┬──────────┐
  │         │ 方向A    │ 方向B    │ 方向C    │
  ├─────────┼──────────┼──────────┼──────────┤
  │ 冲突类型 │          │          │          │
  │ 爽点类型 │          │          │          │
  │ 篇幅     │          │          │          │
  │ 风险     │          │          │          │
  │ 延续性   │          │          │          │
  └─────────┴──────────┴──────────┴──────────┘
```

需要标准示例时，加载 `references/输出模板与示例.md`。

### 模块B：设定扩展

**目标**：基于现有世界观，设计新内容但不破坏已有体系。

**执行步骤**：

```
Step 1: 确认扩展范围
  用户想要什么类型的新设定？
  □ 新地点      □ 新角色    □ 新组织
  □ 新道具      □ 新规则    □ 新机制
  □ 新事件      □ 新关系    □ 其他
  
Step 2: 兼容性检查
  当前世界观有哪些不能打破的约束？
  - 阶段范围（当前角色能接触到什么级别的挑战）
  - 能力/资源体系（角色能力、资源代价、信息来源不能崩）
  - 故事阶段（太早出现高阶内容会破坏节奏）
  - 题材规则（不同题材的核心逻辑不能串台）
  - 方向边界（当前章节服务当前主线，不随手打开无关体系）

Step 3: 设计方案
  给出 2-3 个候选方案，每个包含：
  □ 题材方向标签（本设定服务哪条故事路线）
  □ 设计定位（为什么需要这个设定）
  □ 具体描述（名称/外观/功能/代价/限制）
  □ 与现有体系的关系（兼容性证明）
  □ 在故事中的功能（推动什么剧情）
  □ 正文使用边界（只服务什么玩法，不引出什么无关体系）
  □ 注意事项（使用时的陷阱）
```

需要设定扩展示例时，加载 `references/输出模板与示例.md`。

### 模块C：反转/钩子设计

**目标**：为章尾设计悬念，或为剧情设计合理反转。

**执行步骤**：

```
Step 1: 分析现有伏笔
  当前有哪些未回收的伏笔？
  哪些看似无关的细节可以转化为伏笔？
  
Step 2: 设计钩子（章尾悬念）
  钩子类型：
  □ 信息钩子 —— 新信息突然出现（"纸页背面还有一行字。"）
  □ 冲突钩子 —— 冲突升级/爆发（"门被踹开了。"）
  □ 情感钩子 —— 角色关系突变（"其实我早就见过你。"）
  □ 悬念钩子 —— 未知威胁逼近（"黑暗中有什么东西在移动。"）
  
  每个钩子附带：
  - 一句话钩子
  - 预计在下一章如何展开
  
Step 3: 设计反转（剧情转折）
  好反转的标准：
  ✅ 读者想不到
  ✅ 回头看有伏笔（不是凭空来的）
  ✅ 角色行为合理（不是强行降智）
  
  反转类型：
  □ 身份反转  —— 某人其实是另一势力的人
  □ 动机反转  —— 你以为的好人其实是坏人
  □ 信息反转  —— 之前获得的信息是错的
  □ 结果反转  —— 以为赢了其实是输了（或反过来）
  □ 关系反转  —— 敌人变盟友 / 盟友变敌人
```

需要反转/钩子示例时，加载 `references/输出模板与示例.md`。

### 模块D：灵感整理与组合

**目标**：把零散灵感变成可以比较、可以筛选、可以继续推进的候选方向。

**执行步骤**：

```
Step 1: 判断灵感类型
  □ 题材灵感
  □ 人设灵感
  □ 世界观灵感
  □ 情节钩子
  □ 金手指/能力灵感
  □ 情绪/体验灵感

Step 2: 压缩成一句话假设
  统一改写为：
  "如果……会怎样？"

Step 3: 提炼情感内核
  □ 读者主要满足点是什么？
  □ 是爽、痛、治愈、悬念，还是猎奇？

Step 4: 做候选方向排序
  每个方向至少标：
  □ 扩展空间
  □ 创作难度
  □ 风险提示
  □ 推荐优先级
```

需要完整方法时，加载 `references/公共灵感整理与组合.md`。

## 4. 质量门禁

| 门禁项 | 标准 | 结果 |
|:-------|:-----|:------|
| 方向数量 ≥ 3 | 每次构思至少输出 3 个方向 | 通过 / 不通过 |
| 每个方向有一句话假设 | 不能只给模糊描述，必须能压成一句“如果……会怎样” | 通过 / 不通过 |
| 每个方向有情感内核 | 不能只有设定，没有读者满足点 | 通过 / 不通过 |
| 每个方向有风险提示 | 不能只说好的，必须说出可能的坑 | 通过 / 不通过 |
| 设定兼容性说明 | 新设定必须证明与现有体系兼容 | 通过 / 不通过 |
| 反转有伏笔依据 | 反转不能凭空出现，需标注伏笔来源 | 通过 / 不通过 |
| 不与已有设定矛盾 | 灵感不能违反已建立的世界观约束 | 通过 / 不通过 |

---

## 5. 与审稿 Agent 的协作

创作灵感产生后，进入可行性评估环节：

```
创作 Agent 产出方案
  → 审稿 Agent 介入检查
    □ 设定一致性（与世界观不冲突）
    □ 逻辑合理性（角色行为符合动机）
    □ 体系兼容性（不破坏能力、资源、信息和关系平衡）
  → 产出可行性评估报告
  → 用户结合创作方案 + 评估做最终选择
```

**关键分界**：
- 创作 Agent 负责"能不能更有趣"
- 审稿 Agent 负责"会不会出问题"
- 用户负责"选哪个方向"

---

## 6. 反馈修正机制

当用户对创作灵感的结果不满意时：

1. 记录用户的具体不满（方向不对/不够新/风险没说清）
2. 调整构思策略重新生成
3. 如果同类问题出现 3 次 → 触发规则更新（详见 cases/ 反馈追踪机制）
