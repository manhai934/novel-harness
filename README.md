# Novel-Harness — 长篇网文 AI 创作系统

> 基于 Harness Engineering 方法论构建的长篇网络小说辅助创作系统。
> 目标：支持 100-200 万字量级的连载创作，解决 AI 写作中的遗忘和幻觉问题。
> 核心理念：**写小说不是一个 skill，是一个 Agent 系统。**

## 系统架构

### 四层架构（L0-L3）

```
L0（项目上下文）
   ├── 当前写哪本小说
   ├── 创作约束（成长曲线/防膨胀/阶段天花板）
   └── 文件：.harness/

L1（总编层）
   ├── 理解需求 → 分派任务 → 协调流程
   ├── 不执行具体工作，只做调度和质量管理
   └── 文件：SKILL.md

L2（专业 Agent 层）
   ├── 上下文 Agent — 状态管理 + 信息打包
   ├── 规划 Agent — 剧情构思 + 大纲生成
   ├── 写作 Agent — 正文生成
   ├── 审稿 Agent — 质量审查
   └── 文件：agents/*.md

L3（数据层）
   ├── 项目文件：projects/{项目名}/正文/ 状态/ 记忆/
   ├── 审查规则：game-datafied/ human-linguistics/ plot-review/ rhythm-review/
   ├── 案例库：cases/
   └── **RAG 知识检索层：rag/ (混合检索 + Context Pack)**
```

### 入口与目录边界

当前系统不再把多个独立 skill 直接暴露给用户选择，而是收束为一个总编入口：

- `README.md`：用户入口，说明项目怎么用、目录怎么理解。
- `SKILL.md`：Agent 入口，由总编 Agent 识别需求、分派上下文/规划/写作/审稿任务。
- `.harness/skills/`：Harness 内部能力模块，只由 Agent 或工作流按需调用；暂时保留 `skills` 命名，等旧版 skill 合并完成后再统一评估是否改名。
- `legacy-skills/`：旧版导入 skill 资产，不作为当前主入口；其中有价值的市场扫描、题材分析、写作/审稿资料，后续逐步迁移到 Harness 模块或 RAG 参考库。

这种收束是一次架构升级：用户不需要在大量 skill 中猜该调用哪一个，只需要描述目标，由总编 Agent 负责判断该走哪条流程、加载哪些模块和参考资料。

## 快速开始

### 第一步：初始化 Git

```bash
# 在项目根目录执行
git init
git add .
git commit -m "init: 项目初始化"
```

### 第二步：创建你的小说项目

在 Claude 中调用 novel-harness，Agent 会引导你完成：

```
你：帮我创建新项目
Agent：好的，几个问题：
  1. 书名？
  2. 题材？（数据化降临 / 末世小黑屋 / 其他）
  3. 主角叫什么？
  4. 一句话世界观？

确认后生成：
  projects/{书名}/
  ├── 正文/
  ├── 状态/（由上下文 Agent 自动维护）
  └── 记忆/（由上下文 Agent 自动维护）
```

### 第三步：开始写作

```
"帮我写一章" → 总编 Agent 协调全流程
"帮我审稿"  → 委派审稿 Agent
"没灵感"    → 委派规划 Agent
```

## Agent 体系

系统有 4 个专业 Agent，由总编 Agent 统一协调。每个 Agent 有明确的职责边界。

### 总编 Agent（大管家）

**职责**：理解需求、分派任务、协调流程、质量裁定


| 你说      | 总编做什么                       |
| ------- | --------------------------- |
| "帮我审稿"  | 派审稿 Agent                   |
| "没灵感"   | 派规划 Agent                   |
| "帮我写一章" | 先派上下文 Agent 打包 → 再派写作 Agent |
| "一条龙"   | 规划 → 上下文 → 写作 → 审稿，全流程协调    |


**不做**：不直接审查、不直接写、不自己构思。

### 上下文 Agent（记忆中枢）

**职责**：主角状态追踪、伏笔管理、事件索引、上下文打包

- 写作前：打包当前状态 → 注入写作 Agent
- 写作后：更新所有状态文件 + Git commit
- 维护：主角状态 / 角色状态 / 伏笔登记表 / 事件索引 / 章节摘要

**交付物**：上下文包（结构化文本，含6个信息区）

### 规划 Agent（构思者）

**职责**：剧情构思、大纲生成、设定扩展、反转设计

- 输入：当前故事状态 + 未回收伏笔
- 输出：3-5 个剧情方向 + 对比表
- 原则：不出单一方案，至少给 3 个选择

### 写作 Agent（写手）

**职责**：按大纲和上下文包生成正文

- 输入：上下文包 + 章节大纲
- 输出：2000-3000 字正文
- 约束：遵守项目模板 + human-linguistics 语感规则
- 原则：不自己编剧情、不审查自己的稿子

### 审稿 Agent（质检员）

**职责**：调用各审查子模块，产出统一审查报告

- 审查范围：语感 / 设定 / 情节 / 节奏 / 数值
- 调用子模块：game-datafied / human-linguistics / plot-review / rhythm-review
- 原则：只挑毛病，不写替代方案

## 创作管线

### 单章完整流水线

```
用户确认"写第N章"
  │
  ▼
① 上下文 Agent — 打包当前上下文
   ├─ 主角状态（等级/属性/位置）
   ├─ 出场角色
   ├─ 活跃伏笔（标记超期）
   ├─ 风格参考（最近3章特征）
   └─ 约束提醒（⚠ 不能违反的规则）
  │
  ▼
② 写作 Agent — 生成正文
   ├─ 接收上下文包 + 大纲
   ├─ 写 2000-3000 字
   └─ 遵守 human-linguistics 语感规则
  │
  ▼
③ 上下文 Agent — 写作后归档
   ├─ 更新主角状态
   ├─ 更新伏笔登记表
   ├─ 更新事件索引
   ├─ 生成章节摘要
   └─ git commit
  │
  ▼
④ 审稿 Agent — 质量检查
   ├─ 语感检查（human-linguistics）
   ├─ 情节检查（plot-review）
   ├─ 节奏检查（rhythm-review）
   └─ 题材检查（game-datafied）
  │
  ▼
⑤ 总编 Agent — 汇总呈现
   ├─ 展示修改建议
   └─ 等待用户确认
```

### 200 万字创作周期


| 阶段   | 章节范围      | 核心任务           |
| ---- | --------- | -------------- |
| 第一卷  | 第1-30章    | 建立世界观 + 主角初始成长 |
| 第二卷  | 第31-80章   | 扩大格局 + 引入势力冲突  |
| 第三卷  | 第81-150章  | 深化主线 + 伏笔集中回收  |
| 第四卷  | 第151-250章 | 世界观揭秘 + 格局升级   |
| 第五卷+ | 第251章+    | 终极冲突 + 收束全部线索  |


每卷结束后由规划 Agent + 上下文 Agent 做卷总结和下一卷大纲调整。

## RAG 知识检索层（L3 数据层）

RAG（Retrieval-Augmented Generation）层是 Agent 的"知识外脑"——为写作过程中各类任务提供即时的知识检索服务。

### 解决的问题

- **Agent 遗忘**：长篇写作中，Agent 容易忘记之前用过的规则和知识
- **知识分散**：规则、参考、模板分散在 `.harness/skills/` 各目录，Agent 难以快速定位
- **上下文窗口限制**：无法把所有参考文档塞进 prompt，需要精准检索

### 能力

```
Agent 查询
    │
    ├── "这段太像 AI 写的"     → 去AI味最小修改指南（Top-1）
    ├── "这个大纲能展开吗"      → 大纲质量评估清单
    ├── "写之前该准备什么"      → 章节写前准备清单
    ├── "全民求生选哪条路线"    → 子题材规则入口
    └── "角色状态有没有矛盾"    → 状态一致性补充清单
```

### 快速使用

```bash
# 安装
pip install -r rag/requirements.txt

# 构建索引（首次 + 每次知识源变更后）
python rag/scripts/build_index.py

# 查询
python rag/scripts/query.py "这段太像 AI 写的"

# 启动 HTTP 服务（Agent 通过 API 调用）
python rag/scripts/query.py --server
# → 监听 http://localhost:3456
# → POST /retrieve 检索  |  POST /reindex 重建索引
# → GET /health 健康检查 | GET /stats 索引统计

# 验收
python rag/test/verify.py
```

### 架构要点

| 维度 | 说明 |
|------|------|
| 索引来源 | `.harness/skills/**/rules/*.md`、`references/*.md`、`.harness/projects/*.md` |
| 索引规模 | 23 篇文档 → 233 个语义块 → 233 个向量（384 维） |
| 嵌入引擎 | sentence-transformers（多语言模型，首选）/ TF-IDF 统计嵌入（回退） |
| 存储引擎 | SQLite + FTS5 全文索引 / JSON 文件向量索引 |
| 检索方式 | metadata 硬过滤 → FTS5 关键词召回 → 向量语义召回 → 加权重排 |
| 任务路由 | 7 种写作任务类型自动识别（humanization / outline_review / chapter_prewrite 等） |
| 输出格式 | Context Pack（结构化 JSON + 纯文本，可直接注入 Agent prompt） |
| API 端点 | 6 个（retrieve / reindex / health / stats / routes / explain-retrieval） |

### 更多

详细操作说明见 `rag/OPERATIONS.md`。

## 状态与记忆系统

### 状态文件（由上下文 Agent 自动维护，禁止手动修改）


| 文件            | 内容              | 更新时机   | 读取者            |
| ------------- | --------------- | ------ | -------------- |
| `状态/主角.md`    | 等级/属性/位置/任务/装备  | 每章完成后  | 写作/规划/审稿 Agent |
| `状态/角色/王磊.md` | 配角状态/关系/已知信息    | 该角色出场后 | 写作/审稿 Agent    |
| `状态/伏笔登记表.md` | 编号/内容/埋下章/预计回收章 | 每章完成后  | 规划 Agent       |


### 记忆文件（由上下文 Agent 自动维护）


| 文件               | 内容         | 用途       |
| ---------------- | ---------- | -------- |
| `记忆/事件索引.md`     | 重大事件按章索引   | 快速定位历史事件 |
| `记忆/章节摘要/第N章.md` | 200 字章节概要  | 上下文包素材   |
| `记忆/风格参考.md`     | 最近 5 章写作特征 | 风格一致性保持  |


### 上下文包（每次写作前临时生成）

```
━━ 上下文包 ━━
── 1. 当前任务 ──    ← 本章要写什么
── 2. 主角状态 ──    ← 主角精确的当前状态
── 3. 出场角色 ──    ← 本章涉及的角色
── 4. 活跃伏笔 ──    ← 未回收的伏笔（超期标记）
── 5. 风格参考 ──    ← 最近3章节奏特征
── 6. 约束提醒 ⚠    ← 不能违反的硬约束
```

## 用户分层

系统识别用户的写作经验等级，按不同方式交付反馈：


| 等级            | 特征       | 反馈方式                   |
| ------------- | -------- | ---------------------- |
| **Lv.1 零基础**  | 第一次写小说   | 先夸再批、每次 ≤3 条问题、直接给修改全文 |
| **Lv.2 老书虫**  | 审美在线但手生  | 直言问题、给方向+1 处示范、5-7 条分类 |
| **Lv.3 资深作者** | 有自己的判断标准 | 只标记问题、由作者决策、同行协作姿态     |

## 项目自定义

每个小说项目有自己的创作约束模板，位于 `.harness/projects/`。

### 预置模板


| 模板            | 适用题材          | 特点              |
| ------------- | ------------- | --------------- |
| `模板-数据化降临.md` | 游戏降临 / 副本流    | 数值驱动型成长         |
| `模板-末世小黑屋.md` | 末世 / 安全屋 / 副本 | 非数值驱动、防膨胀、同质化防控 |


### 自定义方法

复制一个模板，修改以下参数即可创建新题材模板：

```
成长曲线：数值驱动 / 非数值驱动 / 混合型
数值膨胀控制：严格 / 一般 / 不控制
同质化防控：开 / 关
阶段天花板：开 / 关
装备系统：替换制 / 改造制
```

## Git 工作流

### 初始化

```bash
# 在项目根目录执行一次
git init
git add .
git commit -m "init: 项目初始化"
```

### 每章完成后的本地记录流程

`projects/` 下的具体创作内容默认不提交到远程。若你只在本地管理小说版本，可以继续用 Git 记录；若要上传某个项目，需要临时调整 `.gitignore` 或手动 `git add -f` 指定文件。

```bash
git add projects/{项目名}/正文/  projects/{项目名}/状态/  projects/{项目名}/记忆/
git commit -m "Ch{N}: {章节标题} — {一句话钩子}"
```

### 关键操作

```bash
# 回溯某章时的主角状态
git show Ch120:projects/{项目名}/状态/主角.md

# 查看两章之间的属性变化
git diff Ch120..Ch130 projects/{项目名}/状态/主角.md

# 实验性分支（测试新剧情方向）
git branch 实验-分支名
git checkout 实验-分支名
# 写几章试试...不满意就切回主线
git checkout main

# 审稿对比（看改了什么）
git diff HEAD~1 projects/{项目名}/正文/第80章.md
```

## 与普通 Skill 的区别


| 维度       | 普通 Skill     | novel-harness             |
| -------- | ------------ | ------------------------- |
| **角色**   | 工具，你调它       | **Agent 系统，它协调你**         |
| **记忆**   | 每次调用零上下文     | **持久化状态 + 伏笔追踪 + 事件索引**   |
| **规模**   | 单次对话         | **100-200 万字连载**          |
| **架构**   | 单文件 SKILL.md | **L0-L3 四层 + 4 专业 Agent** |
| **写作**   | 不支持          | **规划 → 上下文注入 → 写作 → 审查**  |
| **自我进化** | 无            | **反馈追踪 + 3 次误判触发规则更新**    |
| **Git**  | 无            | **每章 commit + 回溯 + 分支实验** |
| **容量**   | 几千字          | **200 万字**                |
