# plot-ideation · 创作灵感 skill

> 本模块是 `qidian-editor` 的**创作子模块**。
> 职责：当作者卡文、需要新剧情、或想探索不同故事走向时，提供多方向的创作灵感。
> 
> **核心理念**：创作 Agent 只管"出点子"，审稿 Agent 负责"挑毛病"。
> 做事的和评判的分开——发散时不自我审查，审查时不发散。

---

## 1. 触发条件

以下情况应加载本 skill：

- **卡文求助**：用户说"不知道接下来怎么写""卡文了"
- **剧情推演**：用户说"如果xxx会怎样""帮我推演一下剧情走向"
- **设定扩展**：用户说"想加个新副本/新怪物/新装备"但没具体想法
- **反转/钩子设计**：用户说"这章结尾加个悬念""帮我设计个反转"
- **备选方案**：用户说"有几个方向拿不准，帮我对比一下"
- **日常灵感**：用户没有具体问题，但想闲聊式碰撞想法

**不适用**：用户已经明确知道要写什么、只是需要执行时。

---

## 2. 执行流程

### 2.1 理解上下文

无论哪种灵感请求，第一步都是理解当前故事状态：

```
加载：
  □ 最新章节的正文或摘要（最近 1-3 章）
  □ 主要角色的当前状态（等级/装备/位置/目标）
  □ 当前已有的伏笔和未回收的线索
  □ 可用资源（队友/道具/信息差）
  □ 世界观约束（什么能做、什么不能做）

产出：context_summary.md
  ── 故事当前位置
  ── 角色状态
  ── 可用伏笔
  ── 世界观边界（不能打破的约束）
```

**关键**：如果用户没有提供上下文，主动问清楚再开始。

### 2.2 判断灵感类型

根据用户需求，进入对应模块：

```
用户说：
  "接下来怎么写？"          → 模块A：剧情构思
  "想加个新副本"            → 模块B：设定扩展
  "章尾加个悬念"            → 模块C：反转/钩子设计
  "有几个方向拿不准"        → 模块A + 模块C（先发散，再配钩子）
  模糊请求                  → 先按模块A走，同时问用户偏好
```

---

### 模块A：剧情构思

**目标**：基于当前故事状态，生成多条可能的剧情走向。

**执行步骤**：

```
Step 1: 识别可用资源
  当前冲突是什么？角色有什么未完成的目标？有什么可用伏笔？
  
Step 2: 发散构思（产出 3-5 个方向）
  每个方向包含：
  □ 核心冲突/爽点类型
  □ 预计篇幅（短期 1-3 章 / 中期 5-10 章 / 长期 10+ 章）
  □ 所需的新角色/场景（如果有）
  □ 风险提示（会不会破坏平衡/会不会拖节奏）
  
Step 3: 对比呈现
  方向A vs 方向B vs 方向C：
  ┌─────────┬──────────┬──────────┬──────────┐
  │         │ 方向A    │ 方向B    │ 方向C    │
  ├─────────┼──────────┼──────────┼──────────┤
  │ 冲突类型 │          │          │          │
  │ 爽点类型 │          │          │          │
  │ 篇幅     │          │          │          │
  │ 风险     │          │          │          │
  │ 延续性   │          │          │          │
  └─────────┴──────────┴──────────┴──────────┘
```

**示例输出**：

```
用户：林北刚拿到双倍奖励，接下来怎么写？

方向A：利用优势快速升级
  - 冲突：战斗爽点，靠双倍经验快速冲级
  - 爽点类型：升级爽点 + 战斗爽点
  - 篇幅：短期（2-3章）
  - 风险：容易变无敌流，爽完就没了
  - 延续性：低——升级完了就得找新目标

方向B：低调发育，关键副本一鸣惊人
  - 冲突：蓄力爆发型，先压抑再释放
  - 爽点类型：装逼爽点 + 收获爽点
  - 篇幅：中期（5-8章）
  - 风险：前期可能太憋屈
  - 延续性：高——一鸣惊人后引发连锁反应

方向C：双倍奖励被发现，卷入阵营冲突
  - 冲突：人际冲突 + 资源争夺
  - 爽点类型：情感爽点 + 战斗爽点
  - 篇幅：长期（10+章）
  - 风险：需要设计有脑子的对手
  - 延续性：极高——可以延伸出多条故事线
```

### 模块B：设定扩展

**目标**：基于现有世界观，设计新内容但不破坏已有体系。

**执行步骤**：

```
Step 1: 确认扩展范围
  用户想要什么类型的新设定？
  □ 新副本      □ 新怪物    □ 新装备
  □ 新技能      □ 新势力    □ 新地点
  □ 新机制      □ 其他
  
Step 2: 兼容性检查
  当前世界观有哪些不能打破的约束？
  - 等级范围（当前角色能接触到什么级别的挑战）
  - 数值体系（攻击/防御/经验公式不能崩）
  - 故事阶段（太早出现高阶内容会破坏节奏）
  - 题材规则（游戏文不能出现修仙文的东西）

Step 3: 设计方案
  给出 2-3 个候选方案，每个包含：
  □ 设计定位（为什么需要这个设定）
  □ 具体描述（名称/外观/能力/数值）
  □ 与现有体系的关系（兼容性证明）
  □ 在故事中的功能（推动什么剧情）
  □ 注意事项（使用时的陷阱）
```

**示例输出**：

```
用户：想加一个新副本，但又不想破坏数值平衡

推荐方案：镜像副本（怪物复制玩家属性）

设计定位：
  - 不需要固定数值，任何等级都能打
  - 考验的是战斗技巧而非硬属性

具体描述：
  - 入口：每个主城都有一个镜像大厅
  - 机制：副本怪物精确复制玩家的等级和属性
  - 奖励：通关后根据"效率分"发放（越快越好）
  - 特殊掉落：镜像碎片（可兑换技能点）

兼容性：
  - 不依赖固定数值 → 任何等级都能打
  - 奖励与投入匹配 → 不会造成数值膨胀
  - 持续可重复 → 是日常副本而非一次性的

故事功能：
  - 让主角展示战斗技巧（别人打不过是因为菜）
  - 可以设置排行榜 → 装逼爽点
  - 技能点兑换 → 长期成长线
```

### 模块C：反转/钩子设计

**目标**：为章尾设计悬念，或为剧情设计合理反转。

**执行步骤**：

```
Step 1: 分析现有伏笔
  当前有哪些未回收的伏笔？
  哪些看似无关的细节可以转化为伏笔？
  
Step 2: 设计钩子（章尾悬念）
  钩子类型：
  □ 信息钩子 —— 新信息突然出现（"面板上多了一行字——"）
  □ 冲突钩子 —— 冲突升级/爆发（"门被踹开了。"）
  □ 情感钩子 —— 角色关系突变（"林北，其实我是——"）
  □ 悬念钩子 —— 未知威胁逼近（"黑暗中有什么东西在移动。"）
  
  每个钩子附带：
  - 一句话钩子
  - 预计在下一章如何展开
  
Step 3: 设计反转（剧情转折）
  好反转的标准：
  ✅ 读者想不到
  ✅ 回头看有伏笔（不是凭空来的）
  ✅ 角色行为合理（不是强行降智）
  
  反转类型：
  □ 身份反转  —— 某人其实是另一势力的人
  □ 动机反转  —— 你以为的好人其实是坏人
  □ 信息反转  —— 之前获得的信息是错的
  □ 结果反转  —— 以为赢了其实是输了（或反过来）
  □ 关系反转  —— 敌人变盟友 / 盟友变敌人
```

**示例输出**：

```
用户：这章结尾想加个悬念

推荐方案：信息钩子

当前情境：
  - 林北刚打完副本，在清点收获
  - 队伍其他人在安全区等他

钩子设计：
  "林北正要退出副本界面——屏幕上突然弹出一条系统消息。
  他看了一眼，手指僵住了。"

下一章展开：
  - 消息内容：一个限时紧急任务
  - 任务奖励无法拒绝
  - 但必须独自前往，不能带队友
  - 制造"去还是不去"的选择冲突

伏笔回收机会：
  - 如果之前埋过"系统任务有隐藏条件"的线索，这里可以联动回收
```

## 4. 质量门禁

| 门禁项 | 标准 | 结果 |
|:-------|:-----|:------|
| 方向数量 ≥ 3 | 每次构思至少输出 3 个方向 | 通过 / 不通过 |
| 每个方向有风险提示 | 不能只说好的，必须说出可能的坑 | 通过 / 不通过 |
| 设定兼容性说明 | 新设定必须证明与现有体系兼容 | 通过 / 不通过 |
| 反转有伏笔依据 | 反转不能凭空出现，需标注伏笔来源 | 通过 / 不通过 |
| 不与已有设定矛盾 | 灵感不能违反已建立的世界观约束 | 通过 / 不通过 |

---

## 5. 与审稿 Agent 的协作

创作灵感产生后，进入可行性评估环节：

```
创作 Agent 产出方案
  → 审稿 Agent 介入检查
    □ 设定一致性（与世界观不冲突）
    □ 逻辑合理性（角色行为符合动机）
    □ 数值兼容性（不破坏平衡）
  → 产出可行性评估报告
  → 用户结合创作方案 + 评估做最终选择
```

**关键分界**：
- 创作 Agent 负责"能不能更有趣"
- 审稿 Agent 负责"会不会出问题"
- 用户负责"选哪个方向"

---

## 6. 反馈修正机制

当用户对创作灵感的结果不满意时：

1. 记录用户的具体不满（方向不对/不够新/风险没说清）
2. 调整构思策略重新生成
3. 如果同类问题出现 3 次 → 触发规则更新（详见 cases/ 反馈追踪机制）
